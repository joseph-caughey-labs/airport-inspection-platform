# Failure mode matrix

Every failure mode the platform is expected to weather, what **detects** it, how it **recovers**, what the **operator sees**, the **logs/metrics** that signal it, whether it's **retryable**, and whether it needs **manual intervention**. Each row is grounded in real code (file paths below), and the honest gaps are called out as _production evolution_ rather than papered over.

Pairs with [ADR 0005 — Idempotency, retries & circuit breakers](adr/0005-idempotency-and-retries.md), the [load + resilience suite](../__TEST__/load/README.md) (which exercises rows 1, 4–7 against the live stack), and the [metrics conventions](operations/metrics.md). Runbooks for the manual-intervention paths live under [`operations/`](operations/) (T-510).

## The health model every row leans on

Two endpoints, one shared meaning across all services:

- **`GET /health`** — liveness. Always `200 {status:"ok"}` if the process is up. Never checks dependencies.
- **`GET /ready`** — readiness. Probes the dependencies this service can't work without and returns `503` when they're down.

| Service                                              | `/ready` checks                                                                                                                                     |
| ---------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| event-pipeline                                       | Redis **and** Postgres ([`app.ts:35`](../services/event-pipeline/src/app.ts))                                                                       |
| sensor-gateway                                       | Redis ([`app.ts:32`](../services/sensor-gateway/src/app.ts))                                                                                        |
| ws-broadcaster                                       | Redis                                                                                                                                               |
| incident-service · audit-service · validation-engine | Postgres ([`app.ts:63`](../services/incident-service/src/app.ts))                                                                                   |
| api-gateway                                          | always ready ([`app.ts:219`](../services/api-gateway/src/app.ts)) — the rate limiter degrades gracefully, so the edge stays up even with Redis down |

docker-compose wires `depends_on: { condition: service_healthy }` against each service's healthcheck, so boot order is enforced at the orchestration layer rather than with in-app wait loops.

## Detail by failure mode

### Ingestion path (1–4, 9, 10)

**1 · Sensor offline.** Today silence is only _felt_ at the edge: the [`WsClient`](../apps/web/composables/useWebSocket.ts) notices the frame gap on disconnect and the dashboard flips its connection chip to `stale`. There is **no server-side heartbeat / `last_seen` watchdog** that raises an alarm when a sensor that should be publishing goes quiet — see _Production evolution_ below. The [sensor-outage e2e scenario](../__TEST__/e2e/scenarios/04-sensor-outage.spec.ts) and [load scenario 04](../__TEST__/load/scenarios/04-redis-outage.scenario.ts) exercise the reconnect/resume half of this.

**2 · AI service down or slow.** AI inference sits on a **separate channel** (`ai.detection.*.emitted`) consumed by the [`AiDetectionBridge`](../services/event-pipeline/src/ai-detections/bridge.ts), not on the sensor-frame hot path — so an AI outage cannot stall ingestion. The bridge fires `void handleMessage(...).catch(...)`, so a slow Postgres insert never blocks the psubscribe loop; malformed envelopes and failed inserts increment `ai_detection_invalid_total{reason}` and are dropped. [Load scenario 06](../__TEST__/load/scenarios/06-ai-timeout.scenario.ts) asserts exactly this isolation. _Gaps:_ no request-timeout/circuit-breaker on a synchronous AI call (there isn't one yet — inference is a stub) and no DLQ for inserts that fail while Postgres is down.

**3 · Duplicate event.** Two layers, per [ADR 0005](adr/0005-idempotency-and-retries.md): an in-memory [`DedupStore`](../services/event-pipeline/src/dedup/store.ts) (5s TTL window) suppresses repeats of the same `idempotency_key` before they hit the handler (`consumer_suppressed_total{queue}`), and the persistence layer's `ON CONFLICT (idempotency_key) DO NOTHING` ([`persist-handler.ts`](../services/event-pipeline/src/persistence/persist-handler.ts)) makes the write itself idempotent for repeats outside the window. Fully automatic, invisible to the operator.

**4 · Queue backlog / overload.** Redis pub/sub doesn't buffer, so "backlog" is the [`ConsumerOrchestrator`](../services/event-pipeline/src/consumers/orchestrator.ts) saturating its in-flight window. At `maxConcurrency` (default 32) it **sheds** — drops the message and counts `consumer_dropped_total{queue}` — rather than growing unboundedly or crashing. ioredis's at-least-once redelivery means shed messages can still arrive later. [Load scenario 03](../__TEST__/load/scenarios/03-queue-backlog.scenario.ts) drives 5 000 frames/s and asserts the service stays alive and keeps processing; drops are the _expected_ graceful-degradation signal.

**9 · Invalid / malformed payload.** `zod` `safeParse` guards every boundary: the [sensor-frames consumer](../services/event-pipeline/src/consumers/sensor-frames.ts) throws on bad JSON / schema violation → `consumer_errors_total{queue}`; REST routes return `400 validation_failed` through the [`@aip/http-safety`](../packages/http-safety/src/index.ts) envelope (4xx keeps the field message, 5xx is scrubbed). Permanent failures — never retried, correctly dropped.

**10 · Out-of-order events.** The [`WatermarkTracker`](../services/event-pipeline/src/prioritization/watermark.ts) keeps a per-`sensor_id` high-water mark (30s tolerance). In-order and `late_in_window` frames pass through; `late_beyond_window` frames go to a bounded [`ReplayQueue`](../services/event-pipeline/src/prioritization/replay-queue.ts) (max 1024) that the [replay worker](../services/event-pipeline/src/replay/worker.ts) drains straight to the persist handler (bypassing the watermark so it can't loop). The watermark is **per-process / in-memory** — a multi-replica deployment needs a Redis-backed watermark (ADR 0005 production evolution). _Gap:_ queue-overflow drops the oldest item without a dedicated metric.

### Infrastructure (5, 6, 11)

**5 · Redis unavailable.** [`createRedis`](../packages/redis-client/src/client.ts) retries with backoff (100ms→5s cap, 20 attempts). Degradation is **deliberately uneven by blast radius**: the api-gateway rate limiter runs `skipOnError: true` (a Redis flap means "no limit applied", not 500s); the [security-events publisher](../packages/security-events/src/index.ts) logs + counts `security_events_publish_failures_total` and **never throws** (a missed audit row is degradation, not a user-facing failure); but the [incident-service publisher](../services/incident-service/src/events/publisher.ts) **does throw** after counting `incident_events_publish_failures_total` — a transition that can't be announced fails the request rather than silently losing the event. The outbox worker retries unpublished rows every tick. [Load scenario 04](../__TEST__/load/scenarios/04-redis-outage.scenario.ts) stops Redis mid-run and asserts services survive + ingestion resumes.

**6 · Postgres unavailable.** [`createPgPool`](../packages/postgres-client/src/client.ts) caps connection (5s) and statement (30s) time so queries fail fast instead of hanging; [`checkHealth`](../packages/postgres-client/src/health.ts) backs `/ready`. The **outbox is the durability anchor**: rows persist with `published_at IS NULL` and resend on a later tick, so a DB blip doesn't lose events. [Load scenario 05](../__TEST__/load/scenarios/05-db-latency.scenario.ts) freezes Postgres (`docker pause`) and asserts the HTTP surfaces stay live and ingestion recovers after unfreeze.

**11 · Partial startup.** The `/health` vs `/ready` split plus compose `depends_on: service_healthy` means a service is absent from the load balancer until its dependencies answer. [`CONSUMERS_DISABLED`](../services/event-pipeline/src/main.ts) can boot the HTTP surface without starting consumers (useful for migrations / debugging). A hard-missing dependency fails fast — `main()` logs the fatal error and exits for the orchestrator to restart.

### Edge & frontend (7, 8)

**7 · WS disconnect / reconnect.** The [`WsClient`](../apps/web/composables/useWebSocket.ts) runs a `connecting→connected→reconnecting` state machine with exponential backoff + jitter ([`ws-reconnect.ts`](../apps/web/utils/ws-reconnect.ts), 250ms→15s) and reconnects with a `?last_event_id=<cursor>` resume cursor; the bearer token is re-read per connect so a refreshed token re-presents correctly. _Production evolution:_ server-side **hydrate-since-cursor** replay is not wired yet — on reconnect the client gets a presence snapshot + live frames, not a gap-fill of everything missed while offline.

**8 · Frontend stale data.** The [system store](../apps/web/stores/system.ts) exposes `connected | stale | disconnected`, and `isOperational` stays true through `stale` so the operator keeps a usable (if degraded) view rather than a blank screen. The [alerts store](../apps/web/stores/alerts.ts) refetches history on mount / reconnect and dedups by id. _Gap:_ there's no automatic "stale after N seconds of silence" timer — staleness today is driven by connection state, not a data-age watchdog.

## Cross-cutting

**Correlation IDs.** [`correlationHook`](../packages/logger/src/fastify-plugin.ts) reads/echoes `x-request-id` + `x-correlation-id`, stamps every log line via `AsyncLocalStorage`, and security/incident event envelopes carry `correlation_id` so a failure can be traced log → event → audit row by one id. _Gap:_ HTTP clients don't yet auto-forward correlation headers downstream, and pub/sub subscribers start a fresh context.

**Durability / DLQ.** The `event_outbox` table is the de-facto retry buffer (`published_at IS NULL` + `attempts` counter, drained by the [outbox worker](../services/event-pipeline/src/persistence/outbox-worker.ts)). A first-class notification-service DLQ + a `notification.replay_dlq` operator action are scoped but not yet built — inspect stuck rows with `SELECT * FROM event_outbox WHERE published_at IS NULL ORDER BY attempts DESC`.

## Production evolution (known gaps, by intent)

These are deliberate scope edges for a demo, each with a clear path forward:

1. **Sensor-silence watchdog** (mode 1) — a `last_seen` heartbeat in sensor-gateway or a "no frames in N s" detector in validation-engine, raising an incident.
2. **AI call timeout + circuit breaker** (mode 2) — once inference is a real synchronous dependency, wrap it in the existing [`CircuitBreaker`](../packages/http-client/src/circuit-breaker.ts).
3. **Replay-overflow metric + DLQ** (modes 2, 10) — count dropped replay-queue items and dead-letter inserts that fail while Postgres is down.
4. **Server-side WS hydration** (mode 7) — gap-fill from the cursor on reconnect, not just a fresh snapshot.
5. **Data-age staleness timer** (mode 8) — flip to `stale` on data age, independent of socket state.
6. **Distributed watermark / breaker state** (mode 10) — Redis-backed so multi-replica deployments order consistently.
